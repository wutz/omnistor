# 硬件分池（Storage Pool）

同一集群的 SNode 硬件不要求同构。集群把 SNode 按**硬件规格**划分为多个 **Storage Pool（存储池）**：池内同构、池间异构，池同时是**放置边界与故障域边界**，放置引擎在池间做负载与容量均衡。

## 为什么分池

| 问题 | 不分池（单一大池） | 分池 |
| --- | --- | --- |
| 混用不同规格硬件 | 纠删条带跨快慢盘，木桶效应拖垮整条带 | 条带只在池内构建，各池发挥各自硬件的全部性能 |
| 分代扩容（新盘更大更快） | 新旧盘混条带，新硬件被拉平到旧水准 | 新一代硬件独立成池，直接兑现代际提升 |
| 故障爆炸半径 | 单盘故障重建牵动全集群 | 故障与重建被限制在池内 |
| 同批次盘相关性故障 | 同批次盘散落各条带，批次缺陷影响面不可控 | 按批次/机型分池，缺陷影响面就是池边界 |

## 池的定义

- **池 = 一组硬件规格一致（或接近）的 SNode**，典型划分依据：介质类型（TLC/QLC/HDD）、盘容量与代际、机型/批次、机房位置。
- 每个池有**介质类别（media class）**标签：`tlc` / `qlc` / `hdd`。同一介质类别可以有多个池（如两代 TLC 各成一池）。
- 纠删组**严格在池内构建**，条带不跨池；池内再按机架/电源划分子故障域，条带成员跨子故障域分布。
- 超大池（千级 SNode）可再切分为多个 zone，纠删与重建流量限制在 zone 内——zone 是池内的规模控制手段，池是硬件与故障的边界。

```
Cluster
├── Pool tlc-gen1   (media=tlc, 2U 24×7.68TB NVMe × 200 台)   ←┐
├── Pool tlc-gen2   (media=tlc, 2U 24×15.36TB NVMe × 300 台)  ←┼─ 主层（元数据+热数据）
├── Pool qlc-a      (media=qlc, 1U 32×61TB NVMe × 100 台)     ←─ 温层
├── Pool hdd-a      (media=hdd, 4U 90×24TB × 400 台)           ←┐
├── Pool hdd-b      (media=hdd, 4U 102×30TB × 200 台)          ←┼─ 冷层
└── 外部对象存储 (S3)                                           ←─ 归档层
```

## 与分层的关系（正交）

- **分层（tiering）解决"数据放哪类介质"**：按温度在 media class 之间移动（TLC → QLC → HDD → S3），见 [tiering.md](../features/tiering.md)。
- **分池（pooling）解决"数据放该类介质的哪一组硬件"**：同一 media class 内多个池时，放置引擎按池的负载与容量选择目标池。
- 元数据放置在**所有 `media=tlc` 的池**上：Bucket 的 extent 申请按各 TLC 池的容量水位与 IOPS 余量加权分配，仍然做到"全部 TLC SSD 参与元数据"（见 [metadata.md](metadata.md)）。

## 池间均衡

放置引擎在**同一 media class 的多个池之间**做两类均衡：

### 写入时均衡（增量）

- 新写入/新 extent 按**加权评分**选池：`score = f(容量水位, IOPS/带宽余量, 磨损均值)`，容量与负载双低者优先。
- 权重周期性刷新（秒级），避免瞬时热点导致的池间倾斜放大。
- 同一对象/文件的相邻 extent 允许落在不同池，池间不需要任何数据关联。

### 后台再平衡（存量）

- 池间水位差超阈值（如 15%）时，后台把存量 extent 从高水位池迁往低水位池——**迁移以 extent 为单位走标准的读-写-切指针流程**，与分层迁移共用一套机制与低优先级 QoS 令牌。
- 新池上线（扩容/换代）时，只均衡到"新写入足以拉平"的程度，避免无谓搬迁；可配置为激进模式（主动搬存量）或保守模式（只靠新写入自然稀释）。
- 池计划退役时，反向操作：停止新写入放置，存量 extent 全量迁出后整池下线。

### 均衡不跨 media class

TLC 池永远不会为了"均衡"把数据迁到 HDD 池——跨 media class 的移动只由分层（温度）驱动，避免均衡逻辑破坏性能预期。

## 故障域语义

- **重建限制在池内**：盘/SNode 故障时，纠删重建的读写流量全部在本池，其他池的前台性能不受影响（纠删与重建机制见 [data-protection.md](data-protection.md)）。
- **池是爆炸半径上界**：整池故障（如同批次盘固件缺陷集中爆发）最多影响本池承载的数据；元数据跨多个 TLC 池分布时，单池故障只影响其上的元数据分片，其余池的 Bucket 正常服务。
- 对可靠性要求更高的场景，支持**跨池副本**（同一 extent 在两个池各放一份纠删条带或副本），以容量换整池级容灾——默认关闭，按存储策略（storage class）选择开启。
- 合规场景可将**租户绑定到专属池**（物理隔离，见 [multitenancy.md](../features/multitenancy.md)），该池不参与其他租户的放置与均衡。

## 配置示例（概念）

```yaml
pools:
  - name: tlc-gen2
    media: tlc
    members: [snode-0301 .. snode-0600]
    fault_domains: rack        # 池内条带跨机架分布
  - name: hdd-a
    media: hdd
    members: [snode-1001 .. snode-1400]
    zones: 4                   # 超大池切 4 个 zone
balancer:
  watermark_diff: 15%          # 池间水位差触发再平衡
  mode: conservative           # conservative | aggressive
```
