# 数据保护：分布式纠删与并行重建

OmniStor 用**分布式 D+P 纠删码**保护数据：条带跨故障域分布、写入永远写新位置（无读-改-写）、重建只针对受影响的条带并由全体 CNode 并行分担。元数据的保护走另一条路（Bucket journal + checkpoint 本身以纠删条带落盘，见 [metadata.md](metadata.md)）。

## D+P 条带模型

- 一个纠删组（条带）由 **D 个数据分片 + P 个校验分片**组成，D+P 个分片**分别落在 P+D 个不同的故障域**——任意条带内没有两个分片同域。
- 条带宽度可配：**D = 4–16，P = 2 或 4**。默认 **D+2**；超大规模或融合部署（计算与存储同机、故障相关性高）建议 **D+4**。
- 条带越宽，校验开销越低（空间效率 = D/(D+P)），读写并行度越高；上限受池内故障域数量约束（故障域数 ≥ D+P+热备余量）。
- **条带严格在池内构建**（见 [pools.md](pools.md)）：池内再按机架/电源划分子故障域，分片跨子故障域分布。

```
Pool tlc-gen2（故障域 = SNode，D=8, P=2）
stripe #k:  d0  d1  d2  d3  d4  d5  d6  d7  p0  p1
            │   │   │   │   │   │   │   │   │   │
snode:      03  17  42  08  55  21  33  47  12  60   ← 10 个不同 SNode
```

## 写路径：永远写新位置

- CNode 在内存中凑满条带（或超时后补零封口），计算校验分片，**一次性写到 10 个全新的位置**——绝不回读旧条带做读-改-写（RMW）。
  - 消除了 RMW 的写放大与"写洞"（部分写坏整条带）风险；
  - 小写先进入 journal 化的写缓冲（TLC extent），聚合成整条带再落盘；
  - 覆盖写 = 写新条带 + Bucket 内原子切换 extent 指针 + 异步回收旧分片（与分层迁移共用同一套指针切换机制，见 [tiering.md](../features/tiering.md)）。
- 条带成员的选择由放置引擎完成：先选池（容量/负载评分），再在池内选 D+P 个故障域（空闲容量加权 + 轮转磨损均衡）。
- 写入通过**租约围栏**防脑裂双写（与 Bucket 同一套任期机制，见 [dase.md](dase.md)）。

## 读路径与降级读

- 正常读：按 extent 指针直读数据分片所在的 SSD，**不经过任何中间节点转发**（DASE）。
- **降级读**：分片所在 SNode 故障或慢（尾延迟）时，读 D 个存活分片重构目标数据——读路径可以**绕开慢盘**，把故障从延迟事件降级为带宽事件。
- 校验分片周期性 scrub，静默损坏在后台被发现并即时修复。

## 故障域

| 层级 | 说明 |
| --- | --- |
| SSD/盘 | 最小重建单位 |
| SNode | **默认故障域**：条带分片跨 SNode 分布 |
| 机架/电源 | 池内子故障域：分片进一步跨机架分布（`fault_domains: rack`） |
| 池 | 故障与重建的**爆炸半径上界**（见 [pools.md](pools.md)） |

## 并行重建

重建是 DASE + 分片元数据红利最大的场景：

1. **只重建受影响的数据**：SNode 故障后，从条带索引直接列出"有分片落在该域"的条带集合——不扫盘、不扫全命名空间（对比 shared-nothing 系统的整盘重建）。空盘故障零重建。
2. **全员并行**：受影响条带均匀分派给所有 CNode，每个 CNode 独立地读 D 个存活分片、重构、写入新故障域。**集群越大，参与者越多，重建越快**——重建时间随集群规模下降而非上升。
3. **风险排序**：丢了 2 个分片的条带（P=2 时已无冗余）优先于丢 1 个分片的条带。先把全集群从"再坏一个就丢数据"的状态中捞出来，再慢慢修剩余的。
4. **重建限制在池内**：重建读写流量不出池，其他池的前台性能不受影响。
5. **前台优先**：重建流量走后台低优先级 QoS 令牌（见 [qos.md](../features/qos.md)）；可配置"降级条带比例超阈值时提升重建优先级"。

### 重建目标位置

- 重建出的分片写入**同池内其他健康故障域**（不能与条带现存分片同域），依赖池内预留的热备容量（如 2 个故障域的等效容量）。
- 故障 SNode 修复归来时，其上旧分片已被替代，直接作为空节点重新纳入放置。

## 元数据的保护

- Bucket 的 B-tree 与 journal 本身作为数据写入 TLC 池，同样以纠删条带落盘——**元数据不需要独立的副本机制**。
- Bucket 单写者 + journal 先行保证一致性；纠删保证持久性；两者正交。
- 单池故障时（元数据跨多个 TLC 池分布），只有落在该池的 Bucket 分片受影响，其余 Bucket 正常服务。

## 与 WEKA 的对照（设计参考）

| 机制 | WEKA NeuralMesh | OmniStor |
| --- | --- | --- |
| 保护方式 | 分布式 D+P（3–16 数据 + 2/4 校验） | 同为 D+P，D=4–16，P=2/4 |
| 写模式 | 写新位置 + journal 元数据 | 同：无 RMW，指针原子切换 |
| 故障域 | 服务器/机架/AZ 多级 | SNode/机架多级，外加**池 = 上界** |
| 重建 | 全集群并行、只修受影响数据、越大越快 | 同，且**限制在池内**（爆炸半径可控） |
| 慢盘处理 | 降级读绕开 | 同 |
