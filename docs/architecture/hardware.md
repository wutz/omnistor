# 硬件方案与 ebox

OmniStor 部署在通用 X86 服务器上，采用类似 Vastdata **ebox** 的标准化高密度机箱方案。

## ebox 概念

- ebox = 一个标准化的物理存储单元（机箱/节点组），是集群的基本构建块。
- 多个 ebox 横向堆叠组成集群，软件层（DASE）将所有 ebox 抽象为一个统一共享池。
- ebox 内部角色（前端网关 / 元数据 / 数据）由软件调度，硬件不绑定固定角色——这正是 DASE "解耦" 的体现。

## 推荐硬件配置

OmniStor 支持两种介质分层组合，覆盖性能型与容量型场景：

### 配置 A：性能容量均衡型（SCM SSD + QLC SSD）

| 介质 | 角色 | 典型用途 |
| --- | --- | --- |
| SCM SSD (Storage Class Memory) | 元数据 + 热数据 | 低延迟元数据索引、热对象缓存 |
| QLC SSD | 主数据池 | 大容量主力存储，温/热数据 |

特点：全闪，延迟低，单 ebox 容量与性能均衡。适合对时延敏感的工作负载。

### 配置 B：大容量经济型（TLC SSD + HDD）

| 介质 | 角色 | 典型用途 |
| --- | --- | --- |
| TLC SSD | 元数据 + 热数据 | 元数据、热对象、写缓存 |
| HDD | 冷/温数据池 | 海量冷数据归档 |

特点：成本最优，HDD 提供大容量底座，TLC 兜底性能。适合归档与大规模对象存储。

## 介质分层策略

放置引擎按**访问频度 + 数据温度**在介质间迁移：

```
写路径:  前端 → (SCM/TLC 缓存) → 异步下沉到 (QLC/HDD)
读路径:  命中缓存直接返回；未命中从下层读并按策略提升
```

详见 [features/tiering.md](../features/tiering.md)。

## 节点角色（逻辑，非物理绑定）

| 逻辑角色 | 职责 | 硬件倾向 |
| --- | --- | --- |
| Frontend / Gateway | 协议接入 (NFS/S3/iSCSI) | CPU 强，网卡多 |
| Metadata Service | 命名空间、索引、配额 | SCM SSD，内存大 |
| Data Service | 对象存储、副本、分层 | QLC/HDD 容量大 |
