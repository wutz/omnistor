# 扩展性：万亿对象 / 1 EiB

OmniStor 设计目标：单集群支撑**万亿（10¹²）对象**与 **1 EiB** 容量。

## 扩展瓶颈分析

| 层 | 瓶颈风险 | 应对 |
| --- | --- | --- |
| Access (前端) | 协议连接数、CPU | 无状态横向扩展，按 QPS 弹性 |
| Metadata | 对象数膨胀、IOPS | 哈希分片 + RAFT，分片再平衡 |
| Data | 容量、带宽 | ebox 横向堆叠，共享一切 |
| 网络 | 东西向流量 | 高带宽组网， locality 优先读 |

## 元数据分片

- 万亿对象无法由单一元数据服务承载 → 按**对象 ID 哈希分片**。
- 每分片独立 RAFT 组，承载 ~十亿级对象，分片数随规模线性增长。
- 分片再平衡：新增节点时按负载迁移分片，迁移期间双写保证一致。
- 目录/桶索引采用前缀分片，支撑大规模 LIST。

## 数据层扩展

- 新 ebox 加入即纳入共享池，DASE 保证新写入自然分散到新节点。
- 容量逼近上限时优先在容量利用率低的 ebox 上放置。
- 纠删组跨 ebox 构建，单 ebox 故障不丢数据。
