# 分层存储与 TLC 池动态分配

OmniStor 的主存储层是统一的 **TLC NVMe 池**（元数据 + 热数据），可选地把冷数据分层到 **QLC NVMe SSD、HDD 或外部对象存储（S3）**。

## 层级模型

```
┌────────────────────────────────────────────────┐
│  主层：TLC NVMe 池（所有 SSD，DASE 共享）        │
│  ├── 元数据（Bucket B-tree + journal）          │←── 动态分配，
│  └── 热数据（新写入 + 高频访问）                 │    无固定边界
└──────────────┬─────────────────────────────────┘
               │ 按温度异步下沉 / 按需提升
   ┌───────────┼──────────────┐
   ▼           ▼              ▼
 QLC NVMe     HDD         外部对象存储 (S3)
 (温, SNode内) (冷, SNode内)  (归档, 集群外)
```

- 三种分层目标可任意组合，也可全不配（纯 TLC 运行）。
- **元数据永不下沉**——10 万亿对象场景下，即使数据在最冷层，元数据仍全部在 TLC，保证 lookup/list 延迟稳定。
- 每个介质类别（media class）内部可有多个硬件池：分层决定数据落到哪类介质，池间均衡决定落到该介质的哪组硬件，二者正交（见 [pools.md](../architecture/pools.md)）。分层迁移与池间再平衡共用 extent 迁移机制与低优先级 QoS 令牌。

## TLC 池动态分配（元数据 ↔ 数据）

元数据与数据共用 TLC 池，不做固定分区：

- 池按**分配单元（extent，如 64 MB）**管理，元数据 Bucket 与数据服务都向同一个分配器申请 extent，释放后归还。元数据容量精确跟随使用量增长（见 [metadata.md](../architecture/metadata.md)）。
- **水位与仲裁**：
  - TLC 池空闲低于**下沉水位**（如 30% 空闲）时，加速把冷数据下沉到下层，为新写入和元数据增长腾空间。
  - 空闲低于**保护水位**（如 10%）时，元数据分配优先于数据写缓存——元数据无处可去（不可下沉），数据可以直写下层。
  - 无下层可下沉（纯 TLC 配置）时，保护水位触发容量告警并按 Quota 拒绝新写。
- **磨损均衡**：分配器在所有 SSD 间轮转放置 extent，元数据/数据混布反而摊平了单盘磨损。

## 放置与迁移策略

- **新写入**：一律先落 TLC（写缓存 + 纠删条带化在 TLC 完成），异步按温度下沉。
- **温度判定**：LFU 窗口 + 时间衰减，后台周期扫描。
- **下沉顺序**：TLC → QLC → HDD → 外部 S3（配置了哪些层就走哪些层，可跳层）。
- **提升（promotion）**：下层读命中按策略提升回 TLC；来自外部 S3 的读默认只进读缓存、不改归属，避免震荡。
- **大对象直写**：超阈值的大冷对象可绕过 TLC 直写下层，避免缓存污染。
- **外部 S3 层**：对象以自包含格式（数据 + 校验 + 加密）写出，元数据仍留在本集群 TLC 池；外部层的可用性故障只影响冷数据读取，不影响命名空间操作。

## 迁移一致性

- 迁移期间旧副本保留，读不中断。
- 迁移完成后原子切换元数据指针（Bucket 内单写者事务），再异步回收旧副本。
- 迁移失败可回滚，不影响数据可用性。

## 与 QoS/Quota 的协同

- 分层迁移不改变对象归属，配额计数不变。
- 迁移与下沉消耗后台带宽/元数据 op，使用低优先级令牌（见 [qos.md](qos.md)），前台永远优先。
- 外部 S3 层的出入流量可单独设置带宽上限，避免打满外联链路。
